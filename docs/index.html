Collecte rosbag
libre et automatique

FreeFox surveille un dossier, détecte les bags terminés, les file en SQLite persistant, et les envoie vers Google Drive ou un NAS — sans abonnement, sans dépendance cloud.

⚡ Upload resumable 🔒 BLAKE3 integrity 📦 .mcap & .db3 🐧 systemd service 🐳 Docker arm64
🦊

Aperçu

FreeFox est un service léger conçu pour les équipes robotiques qui enregistrent des rosbags sur le terrain et ont besoin de les récupérer automatiquement — peu importe où se trouve le robot.

L'alternative la plus proche est la plateforme Foxglove Data, mais cette fonctionnalité est payante. FreeFox est la version libre : même flux, zéro abonnement.

Robot  →  ros2 bag record  →  /bags/  →  FreeFox  →  Google Drive / rsync
💡
FreeFox ne touche pas à ROS 2 lui-même. Il surveille simplement un dossier et réagit aux fichiers qui y apparaissent. Il fonctionne en parallèle de n'importe quel setup ROS 2 existant.
✦

Fonctionnalités

⚡
Upload resumable
Envoi par chunks avec reprise automatique. Les sessions Google Drive resumables sont stockées en SQLite — une coupure réseau n'annule pas le transfert.
🗄
File persistante SQLite
Chaque bag passe par une file d'attente sur disque. Un crash ou un redémarrage ne perd aucune entrée. Les uploads interrompus reprennent automatiquement.
🔒
Intégrité BLAKE3
Empreinte calculée localement, stockée en SQLite et dans les métadonnées Drive. Le dashboard affiche OK, NO ou attente pour chaque transfert.
🔁
Déduplication
Un contenu déjà présent avec le même hash BLAKE3 n'est pas renvoyé. Économise bande passante et quota Drive sur les reboots répétés.
👁
Détection propre
FreeFox attend que la taille du fichier soit stable avant de lancer l'upload. Compatible splits durée, taille, ou arrêt manuel.
📊
Dashboard local
Interface web sur localhost:8765. Progression, débit, ETA, fichiers surveillés, incidents — accessible depuis tout PC du réseau.
☁
Multi-backends
Google Drive par défaut. Rsync pour NAS local, serveur SSH ou dossier monté. Interface backend abstraite — S3 prévu.
🐳
Docker multi-arch
Image publiée sur GHCR pour linux/amd64 et linux/arm64. Déploiement progressif sans remplacer systemd.
⬡

Architecture

FreeFox est composé de quatre couches indépendantes, chacune avec une responsabilité unique.

🤖 Robot — ros2 bag record
fichier .mcap / .db3
▼
👁 File Watcher — inotify + polling
fichier stable détecté
▼
🗄 Upload Queue — SQLite persistant
N workers
▼
⚙ Upload Worker Pool — retry + backoff
HTTPS / rsync
▼
☁ Google Drive
📡 Rsync / NAS
FichierRôle
config.pyLit le YAML, expose des dataclasses typées à tout le reste
watcher.pySurveille un dossier, callback quand un bag est stable
queue.pyFile SQLite : pending / queued / uploading / done / failed
worker.pyPool de threads qui drainenet la queue et uploadent
backends/gdrive.pyUpload résumable chunked vers Google Drive
backends/rsync.pyEnvoi via rsync local ou SSH
service.pyOrchestre tout, gère SIGTERM/SIGINT, CLI

Démarrage rapide

1
Cloner et installer
git clone https://github.com/thoukam/freefox
cd freefox
python3 -m venv .venv
.venv/bin/pip install -e ".[dev]"
2
Configurer
cp config/config.example.yaml config/local.yaml
nano config/local.yaml  # renseigner watch.directory et drive.target_folder_id
3
Lancer le dashboard local
.venv/bin/python scripts/dashboard.py \
  --config config/local.yaml \
  --host 127.0.0.1 --port 8765

Puis ouvrir http://127.0.0.1:8765

4
Accès depuis un autre PC du réseau
.venv/bin/python scripts/dashboard.py \
  --config config/local.yaml \
  --host 0.0.0.0 --port 8765

Depuis le PC client : http://IP_DU_ROBOT:8765

Service systemd

Pour un déploiement permanent sur le robot, FreeFox s'installe comme service systemd. Il démarre au boot, redémarre automatiquement, et draine les uploads en cours avant de s'arrêter.

sudo ./install.sh
sudo nano /etc/freefox/config.yaml
sudo systemctl enable --now freefox
journalctl -u freefox -f
✅
install.sh installe le package Python, copie la config d'exemple dans /etc/freefox/, et enregistre systemd/freefox.service.

Commandes utiles

# Statut
systemctl status freefox
# Logs temps réel
journalctl -u freefox -f
# Mise à jour
pip install --upgrade freefox && sudo systemctl restart freefox

Backend Google Drive

Deux modes d'authentification selon le contexte.

OAuth2 Desktop
Pour tester avec un Drive personnel. Lance un navigateur au premier run. Token mis en cache localement.
drive:
  credentials_file: secrets/client.json
Compte de service
Pour un déploiement robot ou flotte avec Google Workspace. Aucune interaction humaine requise.
drive:
  credentials_file: secrets/sa.json
  target_folder_id: "1ABC..."
📖
Procédure détaillée dans docs/google-drive-setup.md : création projet GCP, activation API Drive, partage du dossier.

Variables d'environnement

FREEFOX_ROBOT_ID=robot-01
FREEFOX_CREDENTIALS=/etc/freefox/credentials.json
FREEFOX_TOKEN_PATH=/var/lib/freefox/token.json
📡

Backend Rsync / NAS

Rsync permet d'envoyer les bags vers un NAS local, un disque monté ou un serveur SSH, sans compte cloud.

Dossier local ou monté

storage:
  backend: rsync
rsync:
  destination: /mnt/nas/freefox

Serveur SSH

storage:
  backend: rsync
rsync:
  destination: user@serveur:/data/freefox
  ssh_command: ssh
🔒
Avec BLAKE3 activé, FreeFox crée un sidecar <filename>.blake3 côté destination pour vérifier les doublons sans relire le bag distant.
📋

Toute la configuration se fait dans un fichier YAML. Voir config/config.example.yaml.

Dashboard local

Interface web légère intégrée. Aucune dépendance supplémentaire.

.venv/bin/python scripts/dashboard.py \
  --config config/local.yaml \
  --host 127.0.0.1 --port 8765 \
  --open  # ouvre le navigateur automatiquement
Transfers
ID, statut, progression, chemin distant, taille, débit, ETA, retries.
Watched Files
Tous les bags dans le dossier : état writing / queued / done, taille, date.
Incidents
Dossier manquant, credentials absents, transfers échoués, bags non enfilés.
Configuration
Rappel des paramètres actifs : robot_id, dossier, SQLite, extensions.
